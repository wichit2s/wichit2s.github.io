โจทย์ชุดที่ 1:
Divide and Conquer Strategy

"แบ่งแยกเพื่อเอาชนะ: จากการเรียงลำดับ สู่การค้นหาคำตอบที่ดีที่สุด"

กลยุทธ์การลดขนาดปัญหาจาก \(O(n^2)\) สู่ \(O(n \log n)\) ที่พบบ่อยที่สุดในข้อสอบ TOI

Core Principle: Divide, Conquer, and Combine

1. Divide

แบ่งปัญหาใหญ่ให้เป็นปัญหาย่อย (Subproblems) ที่มีลักษณะเหมือนเดิมแต่ขนาดเล็กลง

2. Conquer

แก้ปัญหาย่อยด้วยการใช้ Recursion จนกระทั่งถึง Base Case ที่ปัญหาเล็กพอจะตอบได้ทันที

3. Combine

รวบรวมคำตอบจากปัญหาย่อยเข้าด้วยกันเพื่อให้ได้คำตอบของปัญหาเดิม

Note: ในระดับ TOI เรามักใช้ D&C เพื่อเปลี่ยนอัลกอริทึมแบบ Brute Force ให้ทำงานได้เร็วขึ้นภายใน Time Limit

Example 1: Counting Inversions

ปัญหา: นับจำนวนคู่ \((i, j)\) ใน Array \(A\) โดยที่ \(i < j\) แต่ \(A[i] > A[j]\)

Input:
Inversions: (3,2), (6,2), (6,5), (8,2), (8,5)
Total: 5

การนับตรงๆ ใช้ \(O(n^2)\) ซึ่งช้าเกินไปสำหรับ \(n = 10^5\) เราจึงใช้ Merge Sort เข้ามาช่วย

D&C Strategy

  • Divide: แบ่ง Array เป็น 2 ส่วนซ้าย-ขวา
  • Conquer: นับ Inversion ในฝั่งซ้าย และฝั่งขวาแยกกัน
  • Combine: นับ Inversion ระหว่างฝั่งซ้ายและขวา ในขั้นตอนการ Merge

"The Secret is in the Merge Step"

Left: [9-11]   Right: [12, 14]

1. เปรียบเทียบ 1 กับ 2 -> เลือก 1
2. เปรียบเทียบ 3 กับ 2 -> เลือก 2 จากฝั่งขวา!
เนื่องจาก Left เรียงลำดับแล้ว, ถ้า 2 < 3
แสดงว่า 2 < ทุกตัวที่เหลือใน Left ([10, 11])
Inversions += (จำนวนตัวที่เหลือใน Left)

ความซับซ้อน (Complexity)

  • Time: \(O(n \log n)\) เท่ากับ Merge Sort
  • Space: \(O(n)\) สำหรับ Auxiliary Array

Implementation Logic (C++)

long long merge(int l, int mid, int r) {
    long long count = 0;
    int i = l, j = mid + 1, k = l;
    while (i <= mid && j <= r) {
        if (A[i] <= A[j]) temp[k++] = A[i++];
        else {
            temp[k++] = A[j++];
            count += (mid - i + 1); // หัวใจสำคัญ 
        }
    }
    // ... copy remaining and return count ...
}

long long solve(int l, int r) {
    if (l >= r) return 0;
    int mid = (l + r) / 2;
    return solve(l, mid) + solve(mid + 1, r) + merge(l, mid, r);
}

Example 2: Binary Search on Answer

"When you can't calculate, but you can check!"

เป็นเทคนิค D&C ที่ใช้ค้นหาค่าที่เหมาะสมที่สุดใน Search Space ของคำตอบ แทนที่จะค้นหาในข้อมูลนำเข้า

ใช้เมื่อไหร่?

  • โจทย์ต้องการหาค่า Min หรือ Max
  • คำตอบมีความต่อเนื่อง (Monotonicity)
  • มีฟังก์ชัน ok(x) ที่ตรวจสอบได้เร็ว (\(O(n)\))

TOI Highlight

มักมาในโจทย์แนว "หาค่าน้อยที่สุดของค่าที่มากที่สุด" (Min-Max Optimization)

โจทย์: Cutting Logs (แนว TOI)

Problem: มีท่อนไม้ \(N\) ท่อน แต่ละท่อนยาว \(L_i\) ต้องการตัดไม้ให้ได้ชิ้นส่วนที่มีความยาว \(X\) เท่ากันทั้งหมดจำนวน \(M\) ชิ้น จงหาความยาว \(X\)ที่มากที่สุดที่เป็นไปได้

Constraint: \(N, M \le 10^5\), ความยาวไม้รวม \(\le 10^9\)

D&C Execution Steps

1

Define Range:

Low = 0, High = \(10^9\) (ค่าความยาวที่เป็นไปได้)

2

The "Check" Function (\(O(n)\)):

ถ้ายาวชิ้นละ \(mid\) จะตัดไม้ได้ทั้งหมด \(\sum \lfloor L_i / mid \rfloor\) ชิ้น ถ้าได้ \(\ge M\) แสดงว่า \(mid\) นี้ใช้ได้

3

Binary Search Loop (\(O(\log(\text{Range}))\)):

ถ้าใช้ได้ -> ลองเพิ่มความยาว (Low = mid) | ถ้าไม่ได้ -> ต้องลดความยาว (High = mid - 1)

Divide and Conquer Checklists

Sorting-based D&C

  • ✓ ใช้ Merge Sort เป็นโครงร่าง
  • ✓ เหมาะกับโจทย์นับคู่/หาความสัมพันธ์ \(i, j\)
  • ✓ Time Complexity มักเป็น \(O(n \log n)\)

Search-based D&C

  • ✓ หาคำตอบที่ Min/Max
  • ✓ สร้างฟังก์ชัน Check ที่เป็น \(O(n)\) หรือ \(O(n \log n)\)
  • ✓ ระวังเรื่อง Integer Overflow ในตัวแปร mid

โจทย์ชุดที่ 2:
Advanced Greedy Strategies

"Beyond Scheduling: กลยุทธ์การเลือกที่พิสูจน์ได้ด้วยเหตุและผล"

การตัดสินใจที่ดูดีที่สุดในระดับย่อย (Local Optimal) เพื่อนำไปสู่คำตอบที่ดีที่สุดระดับโลก (Global Optimal)

Core Concepts

1. No Backtracking

เมื่อตัดสินใจเลือกไปแล้ว จะไม่มีการย้อนกลับมาแก้ไขการเลือกนั้นอีก .

2. Efficiency

มักทำงานได้เร็วมาก (ส่วนใหญ่ติดที่ขั้นตอนการ Sort \(O(n \log n)\)) .

The Challenge in TOI

ความยากคือการหา "Greedy Strategy" ที่ถูกต้องและพิสูจน์ได้ว่าให้คำตอบที่ดีที่สุดเสมอ .

หากพิสูจน์ไม่ได้หรือพบกรณีที่ขัดแย้ง (Counter-example) ควรพิจารณา Dynamic Programming ทันที .

Proof Technique: Swapping Argument

แนวคิดการสลับคู่เพื่อพิสูจน์:

  1. สมมติว่าคำตอบที่ดีที่สุด (Optimal Solution) เลือกสิ่งต่างๆ ในลำดับที่ ต่างจาก Greedy ของเรา .
  2. แสดงให้เห็นว่าหากเรา "สลับ" (Swap) สองสิ่งที่อยู่ติดกันให้ตรงตามแนวทาง Greedy จะทำให้คำตอบ ไม่แย่ลง .
  3. สรุปได้ว่า Greedy ของเราสามารถเปลี่ยนรูปคำตอบใดๆ ให้กลายเป็นคำตอบที่ Optimal ได้เสมอ .

Example 1: Tasks and Deadlines

โจทย์: มีงาน \(n\) อย่าง แต่ละงานใช้เวลาทำ \(d\) (duration) และมีเส้นตาย \(x\) (deadline) คุณจะได้คะแนนแต่ละงานเท่ากับ \(x - (\text{เวลาที่งานเสร็จจริง})\) จงลำดับการทำงานเพื่อให้ได้คะแนนรวมสูงสุด .

ข้อมูลตัวอย่าง (Duration, Deadline):

  • A: (4, 2)
  • B: (3, 5)
  • C: (2, 7)
ควรทำอันไหนก่อนดี?
เส้นตายที่ใกล้สุด?
หรืองานที่สั้นที่สุด?

"Durations are all that matters!"

Greedy Strategy:

เรียงลำดับการทำงานตาม เวลาที่ใช้ทำ (Duration) จากน้อยไปมาก เสมอ .

ทำไมถึงเป็นเช่นนั้น? (Proof Sketch):

หากเรามีงานสองงานติดกัน โดยงานแรกใช้เวลานานกว่างานหลัง (\(a > b\)) เมื่อเราสลับเอางานสั้น (\(b\)) มาทำก่อน:
- งานสั้นจะเสร็จเร็วขึ้น \(a\) หน่วย (คะแนนเพิ่มขึ้น \(a\)) .
- งานยาวจะเสร็จช้าลง \(b\) หน่วย (คะแนนลดลง \(b\)) .
- คะแนนรวมสุทธิจะเปลี่ยนไป \(a - b\) ซึ่งมากกว่า 0 เสมอ! .

Implementation Logic

struct Task { int duration, deadline; };
bool compare(Task a, Task b) {
    return a.duration < b.duration; // หัวใจของ Greedy
}

// ใน main function
sort(tasks.begin(), tasks.end(), compare); // O(n log n) 
long long currentTime = 0, totalScore = 0;
for(auto t : tasks) {
    currentTime += t.duration;
    totalScore += (t.deadline - currentTime); // 
}
cout << totalScore << "\n"; // ระวัง Integer Overflow ใช้ long long

Example 2: Minimizing Sum of Absolute Differences

โจทย์: กำหนดชุดตัวเลข \(a_1, a_2, ..., a_n\) จงหาค่า \\(x\) ที่ทำให้ผลรวมของระยะห่าง \(\sum |a_i - x|\) มีค่าน้อยที่สุด , [15].

ถ้าใช้ค่าเฉลี่ย (Average)?

มักจะพัง! เพราะค่าเฉลี่ยเหมาะกับการลดผลรวมของกำลังสองมากกว่า .

คำตอบที่ถูกต้อง:

The Median

(ค่ามัธยฐาน) .

Mathematical Intuition

พิจารณาเมื่อเราขยับค่า \(x\):
• หากเราเพิ่มค่า \(x\) เล็กน้อย ระยะห่างจากตัวเลขที่ น้อยกว่า \(x\) จะเพิ่มขึ้น .
• แต่ระยะห่างจากตัวเลขที่ มากกว่า \(x\) จะลดลง .

จุดที่สมดุลที่สุดคือจุดที่มี จำนวนตัวเลขฝั่งซ้ายและขวาเท่ากัน ซึ่งนั่นคือความหมายของมัธยฐาน .
Sorted: [1, 2, 2, 6, 9]
Median = 2
Sum = |1-2|+|2-2|+|2-2|+|6-2|+|9-2|
= 1 + 0 + 0 + 4 + 7 = 12 [19].

TOI Competitive Tips

1. Beware of Integer Overflow

ในโจทย์แนว Greedy คะแนนรวมมักมีค่าสูงมากจนเกินขีดจำกัดของ int (32-bit). ให้ใช้ long long (64-bit) เสมอ .

2. Use Standard Libraries

อย่าเขียนฟังก์ชัน Sort เอง! ใช้ std::sort จาก STL เพราะมีประสิทธิภาพ \(O(n \log n)\) และถูกต้องแน่นอน .

"หากคุณต้องเลือกทางเลือก Greedy ที่ซับซ้อนเกินไป ให้ลองหาเคสที่ทำให้มันผิด ถ้าเจอแม้แต่เคสเดียว นั่นคือสัญญาณให้หันไปใช้ DP" .

Greedy Algorithm Summary

  • มองหาคุณสมบัติ Optimal Substructure .
  • พิสูจน์ด้วย Swapping / Exchange Argument .
  • เน้นใช้ Custom Comparators กับ sort หรือ priority_queue .
  • เทคนิคเสริม: Sweep Line สำหรับโจทย์ช่วงเวลาซ้อนกัน .

Contest Strategy & Subtasks

"ไม่ใช่แค่แก้โจทย์ได้ แต่ต้องรู้จักบริหารคะแนนและเวลา"
เป้าหมายคือการทำคะแนนให้ได้มากที่สุดภายใต้ความกดดันและเวลาที่จำกัด

Winning with Subtasks

1. Partial Credit is Key

โจทย์ระดับ TOI มักแบ่งคะแนนเป็นส่วนๆ ตามขนาดข้อมูล (N) หรือเงื่อนไขพิเศษ การเก็บ Subtask ง่ายๆ 20-40 คะแนนจากทุกข้อ อาจช่วยให้ได้เหรียญมากกว่าพยายามทำข้อเดียวให้เต็ม 100

2. Analyze Constraints

อ่านข้อจำกัดของแต่ละ Subtask เพื่อเลือก Algorithm ที่เหมาะสม เช่น \(N \le 20\) (Brute Force), \(N \le 10^3\) (\(O(N^2)\)), \(N \le 10^5\) (\(O(N \log N)\))

Strategy Tip

  • • ส่งให้ไว: เมื่อเขียน Algorithm สำหรับ Subtask เสร็จให้ส่งทันที เพื่อยืนยันว่า logic เบื้องต้นถูกต้องและไม่มี Bug ใหญ่
  • • Regression Test: การทำ Subtask ที่ซับซ้อนขึ้นควรใช้ผลลัพธ์จาก Subtask ง่ายๆ เป็นตัวตรวจสอบความถูกต้อง

Time Management

เมื่อไหร่ควรหยุดทำ 100 คะแนน?

หากเหลือเวลา 30-45 นาทีสุดท้าย และ Algorithm เต็ม 100 คะแนนยังแก้ไม่ออก หรือติด Bug ที่หาไม่เจอ:

หยุดดื้อรั้น พยายามหาทางลัดหรือแก้ Bug ที่กินเวลานาน
เปลี่ยนแผน เก็บ Subtasks ง่ายๆ ของข้ออื่นที่ยังไม่ได้แตะเพื่อสะสมคะแนนเพิ่ม

"ในการแข่งจริง 10 คะแนนที่ได้จาก Brute Force มีค่าเท่ากับ 10 คะแนนจาก Algorithm ขั้นสูง"
