What is Divide and Conquer?

อัลกอริทึมที่แก้ปัญหาโดยการแบ่งปัญหาใหญ่ออกเป็นส่วนย่อย ๆ ที่เป็นอิสระต่อกัน

1. Divide แบ่งปัญหาเดิมเป็นปัญหาเดิมที่ขนาดเล็กลง
2. Conquer แก้ปัญหาที่แบ่งออกมา (มักใช้การ Recursion)
3. Combine นำคำตอบย่อยมารวมกันเป็นคำตอบสุดท้าย
หัวใจสำคัญคือ
การแบ่งปัญหาเป็นส่วนที่ไม่ทับซ้อนกัน (Disjoint Parts)

The Base Case in D&C

การแบ่งปัญหาจะทำไปเรื่อย ๆ จนถึงจุดที่ปัญหาเล็กที่สุดจนหาคำตอบได้ทันที

  • หากแบ่งปัญหาจนเหลือข้อมูลเพียง 1 ตัว ปัญหานั้นมักจะถือว่า "แก้สำเร็จแล้ว" (เช่น การเรียงลำดับ)
  • Base case คือจุดสิ้นสุดของการ Recursion เพื่อป้องกัน Stack Overflow
  • ใน CP การออกแบบ Base case ที่ถูกต้องช่วยให้ Logic ของ D&C ทำงานได้อย่างแม่นยำ
เมื่อถึง Base case อัลกอริทึมจะหยุดแบ่งและเริ่ม
ขั้นตอนการ Conquering และ Combining

Complexity Analysis

เรามักวิเคราะห์ D&C ผ่านสมการความสัมพันธ์เวียนเกิด (Recurrence Relation)

Binary Search T(n) = T(n/2) + O(1)
Merge Sort T(n) = 2T(n/2) + O(n)
Karatsuba T(n) = 3T(n/2) + O(n)
เราสามารถใช้ Master Theorem
เพื่อหาค่า Big O จากสมการเหล่านี้ได้อย่างรวดเร็ว

Pros & Cons of D&C

จุดแข็ง (Pros)

  • แก้ปัญหาที่ซับซ้อนได้อย่างเป็นระบบ
  • ประสิทธิภาพสูง (เช่น log n)
  • มักจะเป็นพื้นฐานของ DS ขั้นสูง (Segment Tree)

จุดอ่อน (Cons)

  • Recursion อาจทำให้เกิด Overhead
  • ใช้หน่วยความจำเพิ่มขึ้น (Stack space)
  • ถ้าปัญหาทับซ้อนกัน จะช้ามาก (ควรไปใช้ DP แทน)
"Dynamic Programming คือ
Recursion ที่ไม่มีการคำนวณซ้ำ (Recursion without repetition)"

Binary Search Essentials

การหาตำแหน่งของค่า x ในข้อมูลที่ เรียงลำดับแล้ว (Sorted)

  • ใช้วิธีแบ่งครึ่งช่วงข้อมูลที่เป็นไปได้ไปเรื่อย ๆ (Halving the interval)
  • หากข้อมูลมี \(n\) ตัว จะใช้เวลาเพียง \(\log_2 n\) ครั้งในการตรวจสอบ
  • ถือเป็นอัลกอริทึมพื้นฐานที่มีประสิทธิภาพสูงสุดตัวหนึ่งใน CP
กฎเหล็กข้อเดียวของ Binary Search คือ
"ข้อมูลต้องเรียงลำดับมาแล้วเท่านั้น"

Why O(log n) matters?

เมื่อขนาดข้อมูล \(n\) เพิ่มขึ้น ความต่างจะชัดเจนมหาศาล

Data Size (\(n\)) Linear (\(n\)) Binary (\(\log n\))
1,000 1,000 ครั้ง 10 ครั้ง
1,000,000 1,000,000 ครั้ง 20 ครั้ง
1,000,000,000 1,000,000,000 ครั้ง 30 ครั้ง
สำหรับ \(n = 10^9\) Binary Search
ทำงานเร็วกว่า Linear Search ประมาณ 33 ล้านเท่า!

The Step-by-Step Logic

  1. กำหนดช่วงค้นหาที่เริ่มต้นจาก low ถึง high
  2. คำนวณหาจุดกึ่งกลาง mid = (low + high) / 2
  3. ตรวจสอบค่าที่ตำแหน่ง mid
    • ถ้า array[mid] == x จบการทำงาน (พบแล้ว)
    • ถ้า array[mid] > x แสดงว่าค่าที่ต้องการต้องอยู่ด้านซ้าย ให้ปรับ high = mid - 1
    • ถ้า array[mid] < x แสดงว่าค่าที่ต้องการต้องอยู่ด้านขวา ให้ปรับ low = mid + 1
  4. ทำซ้ำจนกว่าจะพบ x หรือ low > high
อัลกอริทึมนี้รักษาสิ่งที่เรียกว่า Invariant
คือการยืนยันว่าคำตอบต้องอยู่ในช่วงที่เลือกเสมอ

Implementation in C++

int low = 0, high = n - 1;
while (low <= high) {
    int mid = low + (high - low) / 2;
    if (array[mid] == x) return mid;
    if (array[mid] < x) low = mid + 1;
    else high = mid - 1;
}
return -1; // Not found

Pro Tip: ใช้ low + (high-low)/2 แทน (low+high)/2
เพื่อป้องกันปัญหา Integer Overflow เมื่อตัวเลขมีค่ามาก ๆ

Method 2: Binary Jumps

แทนที่จะลดช่วง เราใช้วิธี "กระโดด" ด้วยระยะทางที่ลดลงทีละครึ่ง 

int k = 0;
for (int b = n/2; b >= 1; b /= 2) {
    while (k+b < n && array[k+b] <= x) k += b;
}
if (array[k] == x) // x found at index k

วิธีนี้เขียนได้สั้นและลดความสับสนเรื่องช่วงเปิด/ปิด (Boundaries)

เทคนิคนี้ช่วยให้เราเข้าใกล้เป้าหมายด้วยความเร็วที่ลดลง
คล้ายกับการเคลื่อนที่ของ Zeno's Paradox

Searching on Answer

ใช้หาค่า "ที่มากที่สุด" หรือ "น้อยที่สุด" ที่เป็นไปได้

  • หากฟังก์ชันมีสมบัติ Monotone (เช่น ถ้า x ใช้ได้ x-1 ก็ต้องใช้ได้)
  • เราสามารถเปลี่ยนโจทย์ "จงหาค่าสูงสุด" เป็น "ค่า x นี้ใช้ได้หรือไม่?"
  • ตัวอย่าง: โจทย์ตัดไส้กรอก (Cutting Hot Dogs)
"มันมักจะง่ายกว่าในการตรวจคำตอบ
แทนที่จะคำนวณหาคำตอบที่ดีที่สุดโดยตรง"

Beware of These Bugs!

1. Off-by-one Error

ระวังเรื่องช่วงเปิด/ปิด (Inclusive vs Exclusive)

2. Floating Point Precision

การหาคำตอบที่เป็นทศนิยมอาจเกิด Loop ไม่รู้จบจากข้อจำกัดของ double

3. Non-Monotonic Function

ห้ามใช้ Binary Search กับข้อมูลที่ไม่เรียงลำดับ หรือฟังก์ชันที่ค่าขึ้น ๆ ลง ๆ

การใช้ Fixed Iterations (เช่น for loop 100 รอบ)
เป็นวิธีที่ปลอดภัยที่สุดสำหรับคำตอบที่เป็นทศนิยม

Binary Search on Answer

เทคนิคการหาคำตอบในโจทย์ Optimization เมื่อเราไม่สามารถคำนวณหาคำตอบที่ดีที่สุดได้โดยตรง แต่สามารถตรวจสอบได้ว่าค่าใดค่าหนึ่ง "ใช้ได้" หรือไม่

"เปลี่ยนจากการหาคำตอบ (Search) เป็นการตรวจสอบคำตอบ (Check)"

ใช้เมื่อปัญหาอยู่ในรูป
"จงหาค่า x ที่น้อยที่สุด/มากที่สุด ที่ยังทำให้เงื่อนไขเป็นจริง"

เงื่อนไขสำคัญ: Monotonicity

ฟังก์ชันการตรวจสอบคำตอบ (Decision Function) ต้องมีสมบัติ ทางเดียว

  • ✔ ถ้าคำตอบ x ใช้ได้ คำตอบที่ "ง่ายกว่า" x (เช่น x+1 หรือ x-1) ต้องใช้ได้เสมอ
  • ตัวอย่าง: [No, No, No, Yes, Yes, Yes, Yes]

เป้าหมายของเราคือการหา จุดเปลี่ยน (Boundary) ระหว่าง No และ Yes

การแปลงโจทย์ Optimization

Optimization

"จงหาความยาวไส้กรอกที่ยาวที่สุดที่จะตัดให้แจกจ่ายได้ครบทุกคน"

Decision (Check)

"ถ้าต้องการไส้กรอกยาว L ซม. จะตัดให้ครบทุกคนได้หรือไม่?"

ความยากจะลดลงมาก เพราะการเขียนฟังก์ชัน check(L) มักใช้เพียงแค่ Greedy หรือ Loop ธรรมดา

การออกแบบฟังก์ชัน check(x)

  • รับค่าพารามิเตอร์ x (ค่าที่เราสุ่มมาตรวจสอบ)
  • ใช้วิธีการที่ง่ายที่สุด (มักเป็น Greedy) ในการหาคำตอบที่ดีที่สุดภายใต้ค่า x นั้น
  • คืนค่าเป็น boolean (true/false)
  • Complexity ของ check(x) จะส่งผลต่อเวลาโดยรวม: \(O(\text{check} \times \log(\text{range}))\)

Binary Search on Answer Logic

low = ค่าที่เป็นไปได้น้อยสุด, high = ค่าที่เป็นไปได้มากสุด
while low <= high:
    mid = (low + high) / 2
    if check(mid):
        ans = mid
        low = mid + 1 (กรณีหาค่าสูงสุด)
    else:
        high = mid - 1

เราจะเก็บค่า ans ล่าสุดที่เป็นจริงไว้ และปรับช่วงค้นหาต่อ

Example: Cutting Hot Dogs

โจทย์: มีแท่งไส้กรอก N แท่ง ยาวต่างกัน ต้องการตัดให้ได้ M ชิ้นที่ยาวเท่ากัน ไส้กรอกที่ยาวที่สุดที่เป็นไปได้คือเท่าใด?

bool check(double L) {
    long long count = 0;
    for (double rod : rods) {
        count += (long long)(rod / L); // หนึ่งแท่งตัดได้กี่ชิ้น
    }
    return count >= M;
}

การจัดการทศนิยม (Precision)

เมื่อคำตอบเป็นทศนิยม (double) การใช้ while(low <= high) จะมีปัญหา

วิธีที่ 1: ใช้ค่า Error ต่ำๆ เช่น while(high - low > 1e-9)
วิธีที่ 2: วนลูปจำนวนรอบที่แน่นอน (Fixed Iterations)

ใน CP แนะนำการวนลูป 100 รอบ เพื่อความแม่นยำสูงสุดและป้องกันการเกิด Infinite Loop

Fixed Iterations Code

double low = 0, high = 1e9;
for (int i = 0; i < 100; i++) {
    double mid = (low + high) / 2.0;
    if (check(mid)) low = mid;
    else high = mid;
}
cout << fixed << setprecision(6) << low;

การวน 100 รอบให้ความละเอียดถึง \(2^{-100}\) ซึ่งเพียงพอสำหรับทุกโจทย์

สัญญาณว่าควรใช้เทคนิคนี้

  • 1. โจทย์ถามหาค่า Min/Max ของผลลัพธ์บางอย่าง
  • 2. คำตอบมีช่วงที่ชัดเจน (เช่น 1 ถึง \(10^{18}\))
  • 3. ยิ่งค่าคำตอบมากขึ้น เงื่อนไขยิ่งมีโอกาสเป็นจริง (หรือเท็จ) มากขึ้น
  • 4. หาทางคำนวณย้อนกลับได้ยาก แต่เขียนฟังก์ชัน "ตรวจสอบ" ได้ง่าย

"เป้าหมายคือหาค่า L ที่มากที่สุดที่ทำให้ \(sum \lfloor a_i / L \rfloor \ge M\)"

ข้อควรระวัง

Integer Overflow: การบวก (low + high) อาจเกินค่าที่ int รับได้ ให้ใช้ long long หรือ low + (high-low)/2

Bounds: ตั้งค่า high ให้สูงพอครอบคลุมคำตอบที่เป็นไปได้ แต่ไม่สูงเกินไปจนคำนวณในฟังก์ชัน check ผิดพลาด

Non-Monotone: ตรวจสอบให้มั่นใจว่าฟังก์ชันเป็นทางเดียวจริงๆ หากค่าผลลัพธ์มีลักษณะขึ้นๆ ลงๆ จะใช้เทคนิคนี้ไม่ได้

Merge Sort

อัลกอริทึมการเรียงลำดับที่มีประสิทธิภาพสูง โดยใช้แนวคิด Divide and Conquer

  • คิดค้นโดย John von Neumann ในปี 1945
  • ใช้การแตกปัญหาออกเป็นส่วนย่อยๆ แล้วนำกลับมารวมกัน
  • การันตีความเร็วที่ \(O(n \log n)\) เสมอ ไม่ว่าข้อมูลจะมาในรูปแบบใด

3 ขั้นตอนของ Merge Sort

1. Divide แบ่งอาเรย์ออกเป็น 2 ส่วนที่จุดกึ่งกลาง
2. Conquer เรียงลำดับอาเรย์ย่อยทั้ง 2 ส่วน (ทำซ้ำด้วย Recursion)
3. Combine นำอาเรย์ย่อยที่เรียงแล้ว 2 ส่วนมา "Merge" กัน

กระบวนการแบ่งแยก (Divide)

[1-7]

[1-4] [5-7]
[1, 2] [3, 4] [5, 6] [7]
แบ่งไปเรื่อยๆ จนเหลือตัวเดียว!

Base Case: เมื่อไหร่จะหยุด?

เมื่อแบ่งจนอาเรย์มีสมาชิกเพียง 1 ตัว

  • อาเรย์ที่มีสมาชิก 1 ตัว ถือว่า "เรียงลำดับแล้ว" โดยปริยาย
  • นี่คือจุดที่การ Recursion จะหยุดและเริ่มส่งค่ากลับขึ้นไปเพื่อรวมกัน

ขั้นตอนการ Merge

หลักการ:

เมื่อมีอาเรย์ 2 ส่วนที่เรียงลำดับแล้ว (P1 และ P2) ค่าที่น้อยที่สุดของอาเรย์รวม ต้อง อยู่ที่จุดเริ่มต้นของ P1 หรือ P2 เท่านั้น

เปรียบเทียบค่าหน้าสุดของทั้งสองฝั่ง -> เลือกค่าที่น้อยกว่าใส่ผลลัพธ์ -> เลื่อนตำแหน่ง -> ทำซ้ำ

ตัวอย่างการ Merge

ฝั่งซ้าย (P1)

[1, 2, 8]

ฝั่งขวา (P2)

[6, 7, 9]

ผลลัพธ์ที่ได้ทีละขั้น:

[1, 2, 6-9]

ทำไมถึงเป็น O(n log n)?

  • \(\log n\) levels จำนวนครั้งที่เราแบ่งครึ่งข้อมูลได้คือ \(\log_2 n\) ครั้ง
  • \(O(n)\) work per level ในแต่ละระดับ (Level) ของการ Recursion ขั้นตอนการ Merge ข้อมูลทั้งหมดจะใช้เวลารวมเป็นเส้นตรง (Linear Time)
  • รวมกัน \(O(n \times \log n)\)

เร็วกว่า Bubble Sort (\(O(n^2)\)) อย่างมหาศาลเมื่อ \(n\) มีค่ามาก

สมการความสัมพันธ์เวียนเกิด

T(n) = 2T(n/2) + O(n)

- 2T(n/2) การเรียกตัวเอง 2 ครั้ง โดยแต่ละครั้งขนาดข้อมูลลดลงครึ่งหนึ่ง

- O(n) เวลาที่ใช้ในการ Merge อาเรย์ย่อยเข้าด้วยกัน

Merge Sort Structure

void mergeSort(int a, int b) {
    if (a == b) return; // Base Case
    int mid = (a + b) / 2;
    
    mergeSort(a, mid);     // Sort left half
    mergeSort(mid + 1, b); // Sort right half
    
    merge(a, mid, b);      // Combine
}

ฟังก์ชัน merge จะใช้อาเรย์สำรอง (Temporary Array) ในการช่วยจัดเรียง

Inversion Counting

Inversion คู่ \((i, j)\) ที่ \(i < j\) แต่ \(A[i] > A[j]\) (ค่าที่อยู่ก่อนดันมากกว่าค่าหลัง)

  • เราสามารถดัดแปลง Merge Sort เพื่อ "นับ" จำนวน Inversion ได้ในเวลา \(O(n \log n)\)
  • ในขั้นตอนการ Merge หากเราเลือกค่าจากฝั่งขวามาใส่ก่อน แสดงว่าค่าฝั่งซ้ายที่เหลือทั้งหมดเป็น Inversion กับตัวนี้!

Merge Sort is Stable

Stable Sort คือการที่ข้อมูลที่มีค่าเท่ากัน จะรักษาลำดับเดิมก่อนการเรียงเอาไว้

Merge Sort มีคุณสมบัตินี้ เพราะในการ Merge หากค่าเท่ากัน เราจะเลือกจากฝั่งซ้าย (ซึ่งมาก่อน) มาใส่ก่อนเสมอ

ใน C++ STL ฟังก์ชัน std::stable_sort มักถูกออกแบบโดยใช้ Merge Sort เป็นพื้นฐาน

Merge Sort Summary

จุดเด่น (Pros)

  • \(O(n \log n)\) แม้ในกรณีที่แย่ที่สุด
  • ทำงานได้ดีกับ Linked Lists
  • เป็น Stable Sort

จุดด้อย (Cons)

  • ใช้หน่วยความจำเพิ่ม \(O(n)\) เพื่อเก็บอาเรย์สำรอง
  • ช้ากว่า Quick Sort ในงานทั่วไปเล็กน้อยเนื่องจากค่าคงที่ในการ Copy ข้อมูล

Quick Sort

อัลกอริทึมการเรียงลำดับแบบ Divide and Conquer ที่เร็วที่สุดในทางปฏิบัติ

  • ใช้วิธีการเลือกค่า Pivot (ค่าอ้างอิง) เพื่อแบ่งกลุ่มข้อมูล
  • จัดเรียงข้อมูลในอาเรย์โดยตรง (In-place sorting) ไม่ต้องใช้อาเรย์สำรองเหมือน Merge Sort
  • เป็นพื้นฐานของฟังก์ชัน std::sort ในภาษา C++

หัวใจสำคัญ: การ Partition

เมื่อเลือก Pivot ขึ้นมาหนึ่งตัว เราจะจัดเรียงอาเรย์ใหม่ให้:

ฝั่งซ้าย: สมาชิกทุกตัวต้อง น้อยกว่าหรือเท่ากับ Pivot
ฝั่งขวา: สมาชิกทุกตัวต้อง มากกว่าหรือเท่ากับ Pivot

"เมื่อทำเสร็จ Pivot จะไปอยู่ในตำแหน่งที่ถูกต้องในอาเรย์ที่เรียงแล้วทันที"

ขั้นตอนการทำงาน

  1. เลือกสมาชิกหนึ่งตัวในอาเรย์เป็น Pivot (เช่น ตัวแรก, ตัวท้าย หรือสุ่ม)
  2. แบ่งอาเรย์ (Partition) โดยย้ายตัวน้อยไว้ซ้าย ตัวมากไว้ขวา
  3. ทำซ้ำ (Recursively sort) ในส่วนฝั่งซ้ายและฝั่งขวาที่แบ่งออกมา
  4. หยุดเมื่อช่วงข้อมูลเหลือเพียง 0 หรือ 1 ตัว (Base Case)

ประสิทธิภาพของ Quick Sort

Average Case (กรณีทั่วไป) O(n log n)
Worst Case (กรณีแย่ที่สุด) O(n²)

กรณีแย่ที่สุดเกิดจากความโชคร้ายในการเลือก Pivot (เช่น ข้อมูลเรียงมาแล้วแต่เราเลือกตัวแรกสุดเป็น Pivot เสมอ)

การสุ่มเลือก Pivot (Randomized Pivot) ช่วยเลี่ยง O(n²) ได้เกือบ 100%

Quick Select

"ถ้าต้องการหาแค่สมาชิกที่น้อยที่สุดลำดับที่ k (k-th smallest element) จำเป็นต้องเรียงข้อมูลทั้งหมดหรือไม่?"

  • Quick Select คืออัลกอริทึมที่ดัดแปลงจาก Quick Sort เพื่อหาค่าสถิติลำดับ (Order Statistics)
  • แทนที่จะทำ Recursion ทั้งสองฝั่ง เราเลือกทำเพียง "ฝั่งเดียว" ที่มีคำตอบอยู่

ทำไม Quick Select ถึงเร็วกว่า?

ในการหาค่า k-th smallest:

  • Sort then Find: ต้องใช้เวลา \(O(n \log n)\) เสมอ
  • Quick Select: ใช้เวลาเฉลี่ยเพียง \(O(n)\)
เพราะขนาดข้อมูลที่เราต้องสนใจจะลดลงแบบ Geometric Series: \(n + n/2 + n/4 + ... \approx 2n\)

ตรรกะการเลือกฝั่ง (Reduction)

สมมติแบ่ง Partition แล้ว Pivot อยู่ที่ตำแหน่ง pos:

  • 1. ถ้า pos == k: ยินดีด้วย! เราเจอคำตอบแล้ว
  • 2. ถ้า pos > k: คำตอบต้องอยู่ฝั่งซ้ายของ Pivot เท่านั้น
  • 3. ถ้า pos < k: คำตอบต้องอยู่ฝั่งขวาของ Pivot เท่านั้น
// เราเรียกตัวเอง (Recursion) เพียงครั้งเดียวต่อหนึ่งระดับ!

ความสำคัญของการสุ่ม

Quick Select เป็นอัลกอริทึมประเภท Las Vegas:

  • การันตีว่าได้คำตอบที่ถูกต้อง 100% เสมอ
  • แต่เวลาในการทำงานขึ้นอยู่กับการสุ่ม (Randomness)
  • ใน CP การสุ่ม Pivot ช่วยป้องกันการโดนโจทย์ "ดัก" กรณีแย่ที่สุด (Worst-case)
ในทางปฏิบัติ Quick Select เร็วกว่าการ Sort มากสำหรับข้อมูลขนาดใหญ่

nth_element ใน C++

ใน STL มีฟังก์ชันสำเร็จรูปที่ใช้ Quick Select เป็นพื้นฐาน:

nth_element(v.begin(), v.begin() + k, v.end());

หลังจบฟังก์ชัน:

  • ตำแหน่ง v[k] จะเก็บค่าที่ถูกต้องหากอาเรย์ถูกเรียงลำดับ
  • สมาชิกฝั่งซ้ายของ k จะมีค่าน้อยกว่าหรือเท่ากับ v[k]
  • สมาชิกฝั่งขวาของ k จะมีค่ามากกว่าหรือเท่ากับ v[k]

สรุป Quick Algorithms

Quick Sort Sort All (\(O(n \log n)\))
Quick Select Find 1 Element (\(O(n)\))
Memory In-place (\(O(1)\) extra space)
"เร็ว ทรงพลัง และไม่ต้องใช้หน่วยความจำเพิ่ม"

Time Complexity Analysis

การวัด อัตราการเติบโต ของเวลาที่ใช้เมื่อขนาดข้อมูล \(n\) เพิ่มขึ้น

  • เราสนใจพฤติกรรมในระยะยาว (Asymptotic behavior)
  • ใช้สัญลักษณ์ Big O Notation เพื่อบอกขอบเขตบน (Upper bound)
  • มองข้ามค่าคงที่ (Constant factors) และพจน์ที่มีลำดับต่ำกว่า

ระดับความซับซ้อนที่พบบ่อย

\(O(1)\): Constant (สูตรคณิตศาสตร์)
\(O(\log n)\): Logarithmic (Binary Search)
\(O(n)\): Linear (Loop รอบเดียว)
\(O(n \log n)\): Efficient Sort (Merge/Quick)
\(O(n^2)\): Quadratic (Nested Loops)
\(O(2^n)\): Exponential (Subsets)

Master Theorem เบื้องต้น

ใช้สำหรับหา Big O ของ D&C Recurrence ในรูป: \(T(n) = aT(n/b) + O(n^d)\)

1. Merge Sort: \(2T(n/2) + O(n)\)
\(\rightarrow\) ทำงาน \(n\) เท่ากับงานย่อย \(\rightarrow\) \(O(n \log n)\)

2. Binary Search: \(T(n/2) + O(1)\)
\(\rightarrow\) งานย่อยน้อยลงมาก \(\rightarrow\) \(O(\log n)\)

3. Quick Select: \(T(n/2) + O(n)\)
\(\rightarrow\) งานหลักอยู่ที่ \(n\) \(\rightarrow\) \(O(n)\)

กฎ 1 วินาทีในโอลิมปิก

ประมาณ \(10^8\) (100 ล้าน) การทำงานต่อวินาที

\(n \le 10^7\) \(O(n)\)
\(n \le 10^6\) \(O(n \log n)\)
\(n \le 5,000\) \(O(n^2)\)
\(n \le 500\) \(O(n^3)\)
\(n \le 20\) \(O(2^n)\)

ข้อควรระวังในการวิเคราะห์

  • Hidden Constant: บางครั้ง \(O(n)\) ที่ตัวคูณเยอะ อาจช้ากว่า \(O(n \log n)\) ที่ตัวคูณน้อย
  • Memory Latency: การเข้าถึงข้อมูลที่กระจัดกระจายจะช้ากว่าข้อมูลที่เรียงติดกัน
  • Amortized Analysis: บางการทำงานอาจช้าในครั้งเดียวแต่เฉลี่ยแล้วเร็ว (เช่น vector::push_back)
"Big O บอกความสามารถในการขยายขนาด
ไม่ได้บอกเวลาที่ใช้จริงเสมอไป"
